# Conventions

This document covers Go style choices, error handling patterns, logging, naming, file organization, linting, and commit message format used in NGINX Gateway Fabric.

## Go Style

The project follows [Effective Go](https://go.dev/doc/effective_go), [Code Review Comments](https://github.com/golang/go/wiki/CodeReviewComments), and [100 Go Mistakes](https://github.com/teivah/100-go-mistakes) as baseline references. Project-specific conventions build on top of these.

### Key Patterns

- **Accept interfaces, return structs.** Functions accept interface parameters for flexibility and return concrete structs to avoid heap allocations and hidden types.
- **Use `struct{}` for sentinel values.** Empty structs for sets (`map[string]struct{}`) and signal channels (`chan struct{}`), not bools.
- **Consistent line breaks.** When breaking long function definitions, calls, or struct initializations, break after each parameter/argument/field -- never mid-line.
- **Construct slices with known capacity.** Use `make([]T, 0, cap)` when bounds are calculable.
- **No booleans as function parameters.** Use integers, maps, or structs to represent options, since booleans resist future extension.
- **Dependency injection.** Separate high-level policy from low-level detail. Construct dependencies outside the component and inject them.
- **Functional options for optional arguments.** Required arguments are function parameters; optional arguments use the `Option` function pattern or config structs.
- **Do not copy sync entities.** `sync.Mutex` and `sync.Cond` must not be embedded; store pointers to them.
- **Pass pointers down the stack, not up.** Returning pointers causes heap allocations. Return value types when possible.
- **Use contexts virally.** Functions accepting `context.Context` should pass or derive it for all downstream calls. All async public APIs must accept a context.

## Error Handling

- **Inline error handling preferred.** Use `if err := fn(); err != nil {}` rather than separate declaration and check.
- **Preserve error context.** Wrap errors with `fmt.Errorf("...: %w", err)` as they propagate up the stack.
- **Handle errors once.** Do not log an error and then return it. Either handle it or return it, not both.
- **Libraries return errors; callers handle them.** Async libraries communicate errors via channels or callbacks.
- **Panics for unrecoverable or programming errors only.** Examples: invalid scheme registration, wrong type assertions. Always pass an `error` to `panic()`.

## Logging

The project uses structured logging via [logr](https://github.com/go-logr/logr) backed by [zap](https://github.com/uber-go/zap), initialized through controller-runtime helpers.

### Log Levels

| Level | Content | Audience |
|-------|---------|----------|
| `error` | Non-critical errors requiring user attention | User |
| `info` (default) | Operational information | User |
| `debug` (`V(1)`) | Extra detail for troubleshooting | Developer |

### Log Message Guidelines

- Start messages with an uppercase letter
- Do not interpolate dynamic values into the message string; use key-value pairs: `logger.Info("Got response", "status", "ok")`
- Use camelCase for key names
- Use `"resource"` as the key for Kubernetes resources
- Put each key-value pair on a separate line for readability
- Do not log sensitive information (secrets, passwords, etc.)
- Do not log returned errors (they will be handled up the stack)
- Name loggers after their component: `logger.WithName("component-name")`

Default output format is JSON for production; text format is available for development.

See [docs/developer/logging-guidelines.md](/docs/developer/logging-guidelines.md) for the full specification.

## Naming and File Organization

- The project follows standard Go project layout: `cmd/` for entry points, `internal/` for private packages, `apis/` for CRD types
- Package names are lowercase, single-word where possible
- Test files use `_test.go` suffix in the same package (white-box) or `_test` package (black-box for BDD tests)
- Mock files are generated by Counterfeiter and placed alongside the interface they mock (typically in a `fakes/` or `fake_*.go` pattern)
- Go templates for NGINX config generation use `*_template.go` naming in `internal/controller/nginx/config/`

## Linting

The project runs golangci-lint v2 with approximately 50 linters and formatters enabled. Key settings from `.golangci.yml`:

| Setting | Value |
|---------|-------|
| Line length | 120 characters max |
| Cyclomatic complexity | 15 max |
| Locale | US English |
| Field alignment | Enforced via `govet` |
| Import ordering | Local prefix `github.com/nginx/nginx-gateway-fabric` (via `goimports`) |
| Formatters | `gofmt`, `gofumpt`, `goimports` |
| Ginkgo linter | Focused containers forbidden |
| Dot imports | Allowed only for `gomega` and `ginkgo/v2` |

Run the linter: `make lint`

The `revive` linter enforces ~25 rules including: blank-imports, context-as-argument, error-naming, error-return, exported, import-shadowing, var-naming, and more.

## Pre-Commit Hooks

The `.pre-commit-config.yaml` defines 12 hooks that run before each commit:

trailing-whitespace, end-of-file-fixer, check-yaml, gitleaks, golangci-lint, prettier (JS/CSS/MD), markdownlint, yamllint, doctoc (table of contents), helm-docs, shfmt (shell formatting), helm-schema.

Install once: `pre-commit install`

## Commit Message Format

Final commits (after squashing) follow this structure:

```
<One line summary of the change>

Problem: Brief description of the problem. The "why".

Solution: Detailed description of the solution. Describe the analysis and thought
process. Think of a newcomer reading this two years from now.

Testing (optional): Description of tests run to validate the solution.
```

Additional rules:
- **TODO** comments must be resolved before merging
- **FIXME** comments may be merged but must include your username and a link to a GitHub issue: `// FIXME(username): description. Issue #NNN`
- Do not include customer-identifying information in commit messages

## Comment Tags

| Tag | Rule |
|-----|------|
| `TODO` | Must be fixed before merging |
| `FIXME(username)` | Can be merged; must include username and issue link |
